## Known constraints вЂ” Apple Silicon (ARM64) networking

On Apple Silicon Macs, QEMU's VM-to-VM networking options are restricted without root privileges:

| Option | Status | Reason |
|---|---|---|
| VirtualBox private_network | Not available | VirtualBox cannot run x86 VMs on ARM64 |
| QEMU socket multicast | Not functional | macOS does not route multicast between user-space QEMU processes |
| QEMU vmnet-host | Requires root | Homebrew QEMU lacks the `com.apple.vm.networking` entitlement |

**Implemented workaround** вЂ” k3s joins via QEMU NAT gateway (10.0.2.2):

1. `cp-0` gets a second QEMU user NIC (`k8sfwd`) with `hostfwd=tcp:0.0.0.0:6443-:6443`. This maps `host:6443 в†’ cp-0:6443`, making the k3s API reachable at `10.0.2.2:6443` from any VM.
2. Workers install k3s-agent with `K3S_URL=https://10.0.2.2:6443`.
3. After initial auth, k3s advertises its cluster IP (`192.168.56.10`) for the remotedialer proxy tunnel. Workers add an iptables DNAT rule to redirect `192.168.56.10:6443 в†’ 10.0.2.2:6443`.
4. cp-0 runs policy routing (fwmark + conntrack) on `enp0s3` so SYN-ACK replies egress via the k8sfwd NIC rather than eth0 (asymmetric routing fix).

All workarounds are encoded in the Ansible roles and are idempotent.

## Infrastructure deployment

This section is the authoritative step-by-step guide for deploying the platform. It is written to be followed by both a human operator and an AI coding agent working in this repository.

### Prerequisites

| Tool | Minimum version | Install |
|---|---|---|
| Vagrant | 2.3+ | `brew install vagrant` |
| vagrant-qemu plugin | latest | `vagrant plugin install vagrant-qemu` |
| QEMU | 8+ | `brew install qemu` |
| Terraform | 1.5+ | `brew install terraform` |
| Ansible | 2.14+ | `brew install ansible` |
| Python 3 | 3.10+ | built-in on macOS 13+ |

Verify:
```bash
vagrant plugin list | grep qemu
terraform version
ansible --version
```

### Step 1 вЂ” Prepare SSH keys

Terraform injects your public key into each VM and writes the private key path into the Ansible inventory.

```bash
# If you don't have a key pair yet:
ssh-keygen -t rsa -b 4096 -f ~/.ssh/id_rsa -N ""
```

The default `ssh_public_key_path` in `terraform.tfvars` is `~/.ssh/id_rsa.pub`. Vagrant manages its own per-VM keypair at `.vagrant/machines/<name>/qemu/private_key`; the Ansible inventory points to those Vagrant-managed keys.

### Step 2 вЂ” Choose an environment

```bash
cd terraform/environments/dev   # or prod
```

Dev and prod use different VM names, subnets, and SSH port ranges so both can run simultaneously.

### Step 3 вЂ” Initialise Terraform

```bash
terraform init
```

Downloads the `hashicorp/null` and `hashicorp/local` providers. No Vagrant Terraform provider is used вЂ” Vagrant is invoked directly via `local-exec`.

### Step 4 вЂ” Review the plan

```bash
terraform plan
```

Expected resources:
- `local_file.vagrantfile` вЂ” rendered Vagrantfile written to this directory (git-ignored)
- `null_resource.vagrant_up` вЂ” triggers `vagrant up --no-tty`
- `null_resource.vagrant_inventory` вЂ” parses `vagrant ssh-config` and writes Ansible inventory

### Step 5 вЂ” Apply

```bash
terraform apply
```

Type `yes` when prompted. This will:

1. Render the Vagrantfile (from `modules/vm-cluster/templates/Vagrantfile.tpl`)
2. Run `vagrant up` вЂ” 3 QEMU/ARM64 VMs created and provisioned (3вЂ“8 minutes on first run)
3. Parse `vagrant ssh-config` and write `ansible/inventory/enclaive-dev.ini`

On completion, the `next_step` output shows the exact Ansible command for step 6.

### Step 6 вЂ” Bootstrap the cluster with Ansible

From the repo root:

```bash
cd ansible
ansible-playbook -i inventory/enclaive-dev.ini playbooks/bootstrap.yml
```

The playbook runs 4 plays in order:

| Play | Hosts | Role | What it does |
|---|---|---|---|
| 1 | all | `base` | apt update, overlay/br_netfilter modules, sysctl for k8s networking |
| 2 | all | `security` | UFW rules (22/6443/10250/8472/51820), fail2ban, SSH hardening |
| 3 | control_plane | `kubernetes` | containerd, k3s server, QEMU hostfwd policy routing, exports kubeconfig |
| 4 | workers | `kubernetes` | containerd, k3s agent, iptables DNAT workaround for cluster-IP routing |

The `k3s_token` fact is set in Play 3 and consumed by Play 4 via `hostvars`. Plays must run in the same `ansible-playbook` invocation.

The playbook is **idempotent** вЂ” re-running applies only drift.

**Apple Silicon note:** On first deployment, `unattended-upgrades` may hold the apt lock for several minutes after VM boot. The roles detect and wait for the lock to clear automatically.

### Step 7 вЂ” Verify the cluster

SSH into the control-plane and confirm all nodes are Ready:

```bash
ssh -o StrictHostKeyChecking=no \
    -p 50010 \
    -i terraform/environments/dev/.vagrant/machines/enclaive-dev-cp-0/qemu/private_key \
    vagrant@127.0.0.1 \
    "sudo kubectl get nodes -o wide"
```

Expected output:

```
NAME                    STATUS   ROLES                  AGE   VERSION        INTERNAL-IP
enclaive-dev-cp-0       Ready    control-plane,master   ...   v1.32.4+k3s1   192.168.56.10
enclaive-dev-worker-0   Ready    <none>                 ...   v1.32.4+k3s1   192.168.56.11
enclaive-dev-worker-1   Ready    <none>                 ...   v1.32.4+k3s1   192.168.56.12
```

The kubeconfig is also fetched to `ansible/kubeconfig/dev.yaml` during Play 3.

## Tearing down

```bash
cd terraform/environments/dev
vagrant destroy -f          # destroy VMs immediately
terraform destroy           # clean up Terraform state
```

Or just `terraform destroy` вЂ” the destroy provisioner calls `vagrant destroy -f` automatically.

## Re-applying after config changes

Terraform tracks the SHA256 of the rendered Vagrantfile. Changing VM resources in `terraform.tfvars` and re-running `terraform apply` automatically re-triggers `vagrant up` to apply the changes. The Ansible inventory is re-generated after each `vagrant up`.
